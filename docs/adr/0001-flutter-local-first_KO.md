# ADR-0001: Flutter 로컬 우선 아키텍처 (백엔드 없음)

- **상태:** 승인됨
- **날짜:** 2026-03-14

## 맥락

초기 계획에는 NestJS + PostgreSQL 백엔드와 Docker가 포함되어 있었습니다. 시스템 설계 논의 중, 개인 시간 추적 앱에 별도 백엔드 서버가 정당한지 평가했습니다.

## 검토한 옵션

| 옵션 | 장점 | 단점 |
|------|------|------|
| NestJS + PostgreSQL | 완전한 제어, SQL 쿼리, 스키마 마이그레이션 | 개인 앱에 과잉 설계, 이중 저장 + 동기화 복잡성, 인프라 비용 |
| Flutter + Supabase | 관리형 DB, 내장 인증, 서버 유지 불필요 | 온라인 의존성 추가, 여전히 오프라인 지원 필요 |
| Flutter + Isar (로컬 전용) | 가장 단순, 완전 오프라인, 인프라 없음 | 크로스 디바이스 동기화 불가, 데이터가 기기에 종속 |

## 결정

MVP 및 예상 가능한 미래에 Flutter + Isar (로컬 전용). 필요 시(v2.0) 클라우드 백업 전용으로 Supabase.

## 근거

- 데이터 볼륨이 작음 (~3,000 세션/년) — Isar로 충분히 처리 가능
- 타이머는 오프라인에서 작동해야 함 (지하철, 비행기) — 로컬 우선이 이를 보장하는 유일한 옵션
- 백엔드를 추가하면 Flutter 복잡성이 오히려 증가 (이중 저장, 동기화 로직, 충돌 해결, 네트워크 에러 처리)
- 서버 사이드 보호가 필요한 민감한 데이터 없음
- Repository 패턴으로 UI 수정 없이 나중에 Isar를 Supabase로 교체 가능

## 결과

- 멀티 디바이스 동기화 없음 (개인 사용에는 허용 가능)
- MVP에 백업 안전망으로 데이터 내보내기/가져오기 (JSON) 추가
- Google Calendar 연동은 서버 매개 대신 클라이언트 사이드 OAuth 사용
