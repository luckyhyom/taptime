# AI 에이전트 프로젝트에서 문서 언어 선택과 토큰 효율

> 2026-03-17 | Taptime 프로젝트 개발 중 Claude와의 논의 기록

## 배경

Taptime은 Flutter 기반 시간 관리 앱으로, Claude Code(AI 에이전트)와 함께 개발하고 있습니다. 프로젝트의 모든 문서(CLAUDE.md, PLAN.md, ADR 등)를 영어로 작성해왔는데, 그 이유는 **토큰을 절약하기 위해서**였습니다.

하지만 저는 한국인이고 영어에 능통하지 않습니다. 그래서 질문했습니다:

> "토큰도 절약하고 문서도 읽기 좋게 할 방법이 있을까요?"

---

## 논의 과정

### 1차 제안: 한국어로 바꾸되 구조화하기

Claude의 초기 제안은 "영어를 한국어로 바꾸면서 구조화하면 오히려 토큰이 줄어든다"는 것이었습니다.

예시:
```
Before (영어 장문):
- When any planning change occurs, update docs/planning/CHANGELOG_PLANNING.md with:
  - Original (User Input): user's exact words in Korean, quoted
  - Background: why the change was needed (in English)

After (한국어 구조화):
- 기획 변경 시 → CHANGELOG_PLANNING.md 업데이트
  - Original: 사용자 원문 (한국어)
  - Background: 변경 이유
```

### 제 반문: 간결하면 정보가 부족해지지 않나?

Claude는 "간결함 ≠ 정보 부족"이라고 설명했지만, 저는 근거를 요구했습니다.

### 실측 데이터

Claude가 실제로 프로젝트의 모든 .md 파일의 토큰을 측정했습니다:

**매 대화 자동 로드 파일:**
| 파일 | 토큰 |
|------|------|
| CLAUDE.md | 745 |
| architecture.md | 815 |
| pitfalls.md | 587 |
| code-style.md | 332 |
| testing.md | 241 |
| commit-rules.md | 252 |
| **합계** | **2,972** |

**한국어 vs 영어 토큰 비교 (동일 내용):**
- 규칙 설명: 한국어는 영어의 **128.6%**
- 짧은 규칙: **159.1%**
- 아키텍처 규칙: **238.5%**
- 간결한 규칙: **128.6%**

결론: **한국어는 동일 내용 기준 영어보다 1.3~2.4배 토큰을 사용합니다.**

### 구조화의 효과

"영어 장문 vs 한국어 구조화"를 비교하니:
- 영어 장문: 186 tokens
- 한국어 구조화: 156 tokens (16% 감소)

한국어임에도 구조화로 인해 오히려 줄었습니다. 하지만 **같은 구조화를 영어에 적용하면 더 줄어든다**는 점도 확인했습니다.

### 제 추가 질문: 구조화가 에이전트에게 더 효율적인가?

Claude의 솔직한 답변:

> "확실한 근거는 없습니다. LLM은 텍스트를 토큰 단위로 순차 처리하며, 사람처럼 '한눈에 스캔'하지 않습니다. 문장형이든 구조형이든 토큰으로 변환되면 동일하게 처리됩니다. 제가 '에이전트도 구조화된 규칙을 더 정확하게 따른다'고 말한 것은 과장이었습니다."

이 솔직한 인정이 신뢰를 높였습니다.

## 최종 결정: _KO 파일 접미사 방식

### 왜 이 방식?

- 에이전트용 영어 원본은 그대로 유지 (토큰 효율)
- GitHub에서 사람이 읽을 수 있도록 한국어 번역본 별도 제공
- 동일 폴더에 `PRD.md` + `PRD_KO.md` 나란히 배치

### 다른 방식과 비교

조사 결과, 오픈소스 프로젝트들의 3가지 접근법을 확인했습니다:

| 방식 | 사용 프로젝트 | 적합 규모 |
|------|-------------|----------|
| 별도 레포 | React, Vue | 대규모 |
| 언어별 하위 폴더 (docs/ko/) | Kubernetes | 중~대규모 |
| **파일 접미사 (_KO.md)** | **standard.js, node-best-practices** | **소~중규모** |

Taptime은 번역 대상 20개 미만이므로 파일 접미사가 최적.

### 동기화 문제 해결

처음에 Claude가 우려한 동기화 문제(양쪽 모두 업데이트해야 함)는 다음으로 해결:

1. **번역은 사용자 요청 시에만** 일괄 수행 (매 커밋마다 하지 않음)
2. **커밋 해시 추적**: 각 _KO 파일 상단에 원본 커밋 해시 기록
   ```markdown
   <!-- translated from: PRD.md @ commit c197f75 (2026-03-17) -->
   ```
3. **`/translate-docs` 스킬**: 해시 비교로 오래된 번역만 업데이트

## 배운 것들

### 1. 데이터로 확인하라
"한국어가 토큰을 더 먹는다"는 막연한 추측 대신, 실제 프로젝트 파일로 측정하니 정확한 판단이 가능했습니다.

### 2. AI의 주장을 그대로 받아들이지 말라
"구조화하면 에이전트가 더 잘 이해한다"는 Claude의 초기 주장은 근거가 없었습니다. 질문하니 솔직하게 인정했습니다.

### 3. 읽을 수 없는 문서는 의미가 없다
토큰 절약을 위해 영어로 쓴 문서를 정작 프로젝트 소유자가 읽기 어렵다면, 그건 최적화가 아니라 장벽입니다.

### 4. 두 가지를 모두 가질 수 있다
원본(영어, 에이전트용)과 번역(한국어, 사람용)을 분리하면 토큰 효율과 가독성을 모두 확보할 수 있습니다.

## 적용한 구조

```
README.md          ← 에이전트 + 영어 읽는 사람
README_KO.md       ← 한국어 읽는 사람

docs/planning/
  PRD.md           ← 에이전트가 읽음
  PRD_KO.md        ← 사람이 읽음 (커밋 해시 추적)

.claude/rules/     ← 번역 안 함 (에이전트 전용)
docs/references/   ← 번역 안 함 (에이전트 전용)
```

규칙: 에이전트는 `*_KO.md` 파일을 무시합니다. CLAUDE.md에 이 규칙이 명시되어 있습니다.
